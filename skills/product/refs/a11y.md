# Accessibility Audit Rubric (WCAG 2.1 AA)

Apply this inline. Audit UI code/components against WCAG 2.1 Level AA. The four
principles are **POUR**. For exhaustive checklists, ARIA patterns, keyboard and
screen-reader guides, see the `accessibility` skill (`skills/product/accessibility/`).

## Read the target first

Focus on: HTML structure + semantic elements, ARIA attributes/roles, interactive
element patterns, form labels + error handling, color definitions (for contrast).

## POUR checklist

**Perceivable**
- [ ] Images have alt text (descriptive, or `alt=""` for decorative)
- [ ] Color is not the sole information conveyor
- [ ] Text contrast >=4.5:1 (normal), >=3:1 (large text 18pt+ / 14pt bold)
- [ ] UI component / graphical-object contrast >=3:1
- [ ] Content survives 200% zoom / text resize

**Operable**
- [ ] All interactive elements reachable via Tab
- [ ] Tab order follows visual/logical order
- [ ] No keyboard traps; modals have ESC exit + focus trap
- [ ] Visible focus indicators on all interactive elements (`:focus-visible`)
- [ ] No content flashing >3x/second
- [ ] `tabindex` is 0 or -1 (avoid positive values)
- [ ] Skip-to-main link present

**Understandable**
- [ ] `lang` attribute on `<html>`
- [ ] Form errors identify the field and explain the fix
- [ ] Labels describe purpose (not just appearance); inputs have associated `<label>`
- [ ] Consistent navigation + identification
- [ ] No unexpected context changes; required-field indicators
- [ ] Confirmation on destructive actions

**Robust**
- [ ] Valid HTML; no duplicate IDs
- [ ] ARIA only where native semantics are insufficient (first rule of ARIA: don't)
- [ ] ARIA roles/states/properties correctly used; custom widgets expose name/role/value
- [ ] Dynamic content uses `aria-live` (polite/assertive) / `role="status"|"alert"`

## Semantic HTML

Prefer `<header> <nav> <main> <article> <section> <aside> <footer>`; sequential
heading hierarchy (`h1 -> h2 -> h3`, no skips); `<button>` for actions and `<a>`
for navigation; `<form> <label> <fieldset> <legend>` for forms. Never `<div onclick>`.

## Keyboard testing

`Tab` / `Shift+Tab` (move), `Enter` (activate link/button), `Space` (button /
checkbox), `Escape` (close dialog/menu), arrow keys (within tabs/menus/radios).
Verify: reach all interactive elements, focus order = visual order, focus visible,
no traps, skip link works, modals trap focus, arrows navigate menus/tabs.

## Common violations to flag

Missing alt text; insufficient contrast; inputs without labels; missing focus
indicators; keyboard traps; skipped heading levels; buttons with no accessible
name; non-semantic div/span buttons; autoplaying media without controls; timeout
without warning; duplicate IDs; empty links/buttons; tables without headers;
missing `lang`; incorrect ARIA.

## Quick mode

With `--quick`: return only Critical issues; skip the full checklist.

## Tooling

Color contrast: compute the WCAG ratio against the AA thresholds (4.5:1 body,
3:1 large text). For automated scanning, prefer axe DevTools / Lighthouse /
WAVE / pa11y when available.
