# Timing Lens

Additional questions for the five lenses that challenge WHEN to act, not just
what to do. Catches premature optimization, false urgency, and wrong sequencing.

## Lens 1 Additions: Is This Real?

- Is this a "right now" problem, or are we solving for something 6+ months away?
- Is the urgency real, or manufactured by recency bias? (just saw it, so it feels urgent)
- Are we optimizing for the wrong phase? (pre-MVP polish, scaling before traction)
- Would waiting a week give us better information to make this decision?

## Lens 2 Additions: What's Actually Going On?

- Is the sequencing wrong? Should something else happen first?
- Are we blocked on a decision we don't have enough data to make yet?
- Is this a time-sensitive window (security fix, compliance deadline) or self-imposed urgency?
- Are we conflating "this is important" with "this must happen now"?

## Lens 3 Additions: What Else Can We Fix?

- Are there quick wins we should do first that unblock this work?
- Is there prerequisite work that would make this change cheaper later?
- Can we ship a partial solution now and iterate, instead of building the full thing?

## Lens 4 Additions: Should We Rethink?

- Would deferring this 2 weeks let us batch it with related changes?
- Would making this decision incrementally (reversible steps) be safer than
  committing to the full approach now?
- Are we trying to predict the future? Can we build for adaptability instead?

## Lens 5 Additions: Better Way?

- Can we set a time-boxed experiment instead of a permanent decision?
- Can we build a toggle/flag so this is reversible if our timing is wrong?
- Can we validate the assumption that this is urgent before investing?
- Is "do nothing and monitor" a legitimate option?
