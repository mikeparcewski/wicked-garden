Tests the yolo-revoke audit trail. A project advances through clarify into build via `wicked.phase.transitioned`, then a `wicked.crew.yolo_revoked` event lands with `revoked_count=1` and `revoke_reason=SCOPE_INCREASE_WITHOUT_APPROVAL`. The projector resolves the project by matching `payload.project` against `projects.id` and updates `yolo_revoked_count=1` and `last_revoke_reason` on the project row. Phase rows are not affected by the revoke event. The event_log should contain a row with `projection_status=applied` for the yolo_revoked event. This fixture locks the invariant from issue #581 that yolo revocations are durably reflected in the project projection for audit purposes.
